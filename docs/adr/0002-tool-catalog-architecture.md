# Tool catalog as a closed-set, data-driven registry

Per-tool metadata (CLI name, version probe, install command, executable path, settings override, wizard row, Settings row) is currently scattered across roughly fifteen files with one switch or property per tool in each. Adding `deno` for v0.1.0 made the duplication acute. We consolidate around a single declarative catalog: a closed `ToolKind` enum (`Uv, AudioSeparator, Ytdlp, Ffmpeg, Deno`) plus a `Tool` record holding the descriptor, plus a static `ToolCatalog.All`. Adding a tool is a single-row catalog change plus, if it needs a new install pattern, a new strategy case. Rejected: an extension-point design with `ITool` / `IToolInstaller` polymorphism. The tool set is genuinely closed (no plugin story planned), three strategies cover every known case, and the test-doubles that matter (`IProcessRunner`, `ToolStateService`) are already mockable below the catalog.

Three install strategies model the entire space and are encoded as a sealed `InstallStrategy` discriminated union: `ScriptInstall` (runs an upstream installer, used only by `uv`), `UvToolInstall` (runs `uv tool install`, used only by `audio-separator`), and `BundledFetch` (downloads a pinned archive into `BundledBinDir` and verifies its SHA-256, used by `yt-dlp`, `ffmpeg`, `deno`). Strategy is fixed per tool. Cross-platform variation is data within a strategy (`BundledFetch` carries a `Dictionary<PlatformInfo, AssetDescriptor>` for per-OS asset URL plus SHA), not strategy substitution. Rejected: strategy-per-platform (so macOS could switch ffmpeg to `BrewInstall`), because it loses the "user gets exactly what StemForge pinned" reproducibility property and balloons the per-tool decision matrix. Also rejected: bundling our own portable `uv` to eliminate `ScriptInstall` entirely, because it makes us responsible for uv's update cadence and gains little; `uv` is a tool power users actively want on PATH. The catalog is built around the rubric "if a power user might already have this tool or want to use it independently, install on PATH; otherwise bundle to `BundledBinDir`". This is why `yt-dlp` moves from `UvToolInstall` to `BundledFetch` in this refactor: it would shadow a user's existing `yt-dlp` on PATH, and yt-dlp's own `--update-to` mechanism makes a bundled binary self-updating.

[[Variant]] is a sub-concept of `UvToolInstall`, not a top-level Tool property, because only one tool (`audio-separator`) has variants and they exist precisely to model pip extras and extra-index-urls (an artifact of how the Python package is installed). The variant set is per-platform inside the strategy. Variant detection (runtime probe of torch + onnxruntime providers) lives as data on the strategy too, executed via a dedicated `tools/detect_variant.py` rather than an inline C# string. The wizard install step and the Settings tool-overrides section become `ItemsControl` over `ToolRowViewModel`, one row VM per Tool, with `IsVisible` bindings for variant-specific UI on the audio-separator row. `ToolStateService` exposes per-tool observable state via these row VMs (`state.GetRow(kind).IsAvailable`), replacing the five `IsXxxAvailable` named properties; composite gates like `CanDownloadFromUrl` stay as named observables. Settings overrides migrate from five flat `XxxPath` properties on `AppSettings` to a single `Dictionary<string, string?> ToolPathOverrides`, with a one-shot migration on first load that drains the legacy properties into the dictionary. The v0.1.0 install base is small enough that disk-compat risk is negligible.
