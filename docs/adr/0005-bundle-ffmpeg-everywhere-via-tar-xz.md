# Bundle ffmpeg everywhere via tar.xz, with extraction modelled as two orthogonal axes

Linux ffmpeg ships as `.tar.xz` (the John Van Sickle and BtbN static builds), so the bundled-fetch extractor must read xz-compressed tar archives. .NET 11 has a tar reader (`System.Formats.Tar`) but no xz/LZMA decoder, so we add the SharpCompress NuGet package and use it only to peel the xz layer; the resulting tar stream is read by the BCL `TarReader`. We keep the dependency surface minimal: SharpCompress decompresses, the BCL walks entries.

The former `ExtractMode` enum conflated two genuinely independent concerns: the archive *format* (how to decompress the download) and the *layout* (where the target binary lives inside it). That coupling forced a new enum case per combination, and the Linux ffmpeg case (xz-compressed tar, binary under a `bin/` subpath) would have required a fourth case duplicating the existing flatten logic against a different decompressor. We split these into two orthogonal axes on `BundledAsset`: `ArchiveFormat` (`RawBinary`, `Zip`, `TarXz`) and `BundledLayout` (`DownloadIsBinary`, `SingleFileAtRoot`, `FlattenFromBinSubdir`). The fetcher dispatches decompression on the format and entry selection on the layout, so a new format or a new layout is a single case in one switch rather than a combinatorial new mode. The existing Windows assets map cleanly: yt-dlp is `(RawBinary, DownloadIsBinary)`, deno is `(Zip, SingleFileAtRoot)`, Windows ffmpeg is `(Zip, FlattenFromBinSubdir)`. The future Linux ffmpeg asset will be `(TarXz, FlattenFromBinSubdir)`; the extraction capability lands here but the Linux asset entries themselves are deferred to issue #19. SHA-256 verification is unchanged and still runs on the downloaded bytes before any extraction touches disk.
