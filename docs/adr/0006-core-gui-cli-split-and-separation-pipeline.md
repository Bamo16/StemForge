# Split into Core/GUI/CLI and extract a UI-agnostic separation pipeline

StemForge gains a headless command-line mode: running a separation without the GUI, and downloading audio from a URL (with StemForge's metadata/provenance/thumbnail handling) without separating. Today everything lives in one `WinExe` project, and the job orchestration — download, run each [[Preset]], optional drum extraction, keep-source, and provenance tagging — is fused into `JobQueueService`, interleaved line-by-line with `Dispatcher.UIThread.Post` calls and the construction of `JobItemViewModel`. That orchestration is the logic a CLI needs, but it is not callable outside the GUI: with no Avalonia dispatcher running, those posts queue onto a loop that never pumps.

We restructure into three projects in one solution: `StemForge.Core` (library) owns everything UI-agnostic — process running, paths, settings, HTTP clients, the separator driver, the YouTube service, preset discovery, and a new UI-agnostic [[Separation pipeline]] that takes one [[Job]] and drives it to completion, reporting a single `IProgress<JobUpdate>` stream; `StemForge` (GUI `WinExe`) keeps Avalonia, the views, and the view-models, with `JobQueueService` demoted to a thin GUI adapter that maps pipeline updates onto the queue view via a `Progress<T>` captured on the UI thread (which deletes most of the dispatcher marshalling); `StemForge.Cli` (console `Exe`) consumes the same pipeline and renders progress in the terminal. Composition is layered: Core exposes `AddStemForgeCore` and each host adds only its own registrations (view-models for the GUI, command types for the CLI), so the shared service graph has one source of truth. Log handling splits across three layers — ingestion-time noise normalization below the pipeline (shared), structured progress/log *facts* emitted by the pipeline, and per-front-end display *policy* on top (the GUI's hoard-and-flush-on-failure feed vs. the CLI's streaming).

The decisive trade-off was how the CLI gets a console on Windows. The app is `WinExe` (no attached console), so a single-binary CLI would require `AttachConsole(ATTACH_PARENT_PROCESS)` P/Invoke, and — because the parent shell does not wait on a `WinExe` — an animated ANSI progress display would fight the shell prompt for cursor position and look broken. A separate console-subsystem `Exe` owns its console cleanly: ANSI progress works, the shell blocks until exit, no P/Invoke, no prompt interleaving. Since a UI-agnostic pipeline must be extracted regardless (it is also what makes real end-to-end separation testing possible), the marginal cost of a second thin entry point is small, so the console-exe approach was chosen over the single-binary shim. For the CLI's argument parsing and progress rendering we adopt `Spectre.Console` and `Spectre.Console.Cli` rather than `System.CommandLine`, for cohesion between the verb/argument model and the progress display and to avoid System.CommandLine's long preview-API churn and more awkward dependency-injection story; an argument parser is not expensive to swap later, but recording the rejection here pre-empts the recurring "why not the Microsoft one?" question.

Known costs accepted: the restructure touches every namespace's project membership and the test project's references, and the release now ships two executables (see ADR 0007 for the packaging consequence). Scope for the first cut is deliberately narrow — local or URL sources with batching, built-in presets only, and the metadata/provenance/thumbnail handling — and full GUI/CLI feature parity is an explicit non-goal: each front-end may do things the other cannot. A CLI-only user on a machine that never ran the GUI setup is handled by detect-and-fail-loudly for now; a CLI setup equivalent is deferred.
