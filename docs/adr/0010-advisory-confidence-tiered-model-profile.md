# Model profile: an advisory, confidence-tiered resolution of a model's output stems

StemForge resolves a [[Model]]'s output [[Stem]]s *without running it* — the [[Model profile]] — through an ordered set of confidence tiers, and treats the result as **advisory**: it informs stem-aware UI but never gates a separation. The resolver (`ModelProfileResolver`) tries each tier in descending confidence and takes the first that yields stems: (1) the model's own config / benchmark instrument list, tagged `Config`; (2) an architecture default — Demucs is the fixed four-stem set, MDX and VR are a target plus its complement — tagged `ArchitectureDefault`; (3) a filename-derived target as a last resort, tagged `FilenameTarget`. A model that survives all three with no stems is reported `Unknown` rather than failing. Each resolved stem carries the `StemSource` it came from, so the profile records *how sure* StemForge is, not just *what* it guessed. audio-separator remains the source of truth for what a model actually emits.

## Considered options

- **Make the resolved stems authoritative — drive the run from them.** Rejected. The stem list a model emits is decided by audio-separator at separation time from the actual weights and config it loads; the profile is derived from static benchmark data, fixed architecture rules, and filename heuristics, none of which can see the weights. Treating a derived guess as ground truth would let a wrong filename heuristic or a stale benchmark entry silently misroute or block a separation that audio-separator would have run correctly. Keeping the profile advisory means the worst case is a cosmetically wrong stem label in the Models tab, never a refused or mis-shaped run. ADR 0008 already fixes audio-separator as the inference authority; this is the same boundary applied to stem prediction.
- **A single flat resolution with no recorded source.** Rejected. A bare stem list cannot say whether "vocals, instrumental" came from the model's measured benchmark scores or from matching "voc" in the filename — and those are wildly different in trustworthiness. Tagging every stem with its `StemSource` lets the UI mark inferred stems as such, lets a future [[Keep set]] picker weight a config-derived name above a filename guess, and makes the resolver's behaviour assertable tier by tier in tests. The ordering is the design, so it is encoded (`Config` > `ArchitectureDefault` > `FilenameTarget` > `Unknown`) rather than left implicit.
- **Fetch every model's config up front to maximise accuracy.** Rejected. Configs live remotely and most models are never inspected by the user, so eagerly fetching them spends network and startup latency for stems that benchmark data or an architecture default already answers for free. Config-on-demand is therefore lazy and sits behind a seam (`IModelConfigSource`): the resolver only asks for a config when a config-driven (MDXC/roformer) model has no benchmark stems and no cheaper tier applies, and it fetches the **config only, never the weights**. With no source wired (the v0.3.0 default) the resolver degrades to the architecture and filename tiers instead of failing.
- **Report a best-effort guess instead of `Unknown` for unresolvable models.** Rejected. A model absent from benchmark data, with no architecture default and no filename hint, has nothing to honestly say about its stems; inventing a default ("probably vocals") would be a confident-looking lie. `Unknown` is a first-class outcome the UI shows as "no stems known", which is both truthful and the signal that a config fetch could later improve.

## Consequences

- The Models tab surfaces resolved stems for models the benchmark previously listed none for, marked **inferred** to distinguish an advisory name from a benchmark-measured one (which still shows its SDR). The stem filter matches on resolved profile stems too, so a previously-unknown vocals model now appears under the Vocals filter.
- The confidence tiers are unit-tested in isolation (`ModelProfileResolverTests`), one test per tier asserting both the stem names and the `StemSource` tag, plus that config-on-demand is never hit when benchmark stems exist and is hit exactly once when they do not. The network seam is exercised with a fake (`FakeModelConfigSource`); no test fetches a real config or weight.
- `IModelConfigSource` is the single extension point for live config fetching. Adding real on-demand fetching is a matter of implementing it (a torch-free config-only one-shot, consistent with `list_models.py`) and registering it; the resolver and UI need no change. Until then MDXC models without benchmark stems fall through to the filename tier.
- Because the profile is advisory, a wrong or missing stem prediction is a display issue, not a correctness one: the [[Separation run]] is unaffected. This keeps the heuristics free to improve over time without risk to the pipeline.
