# Self-contained single-file publish per executable

ADR 0006 introduces a second executable (`StemForge.Cli`) alongside the GUI, both sharing `StemForge.Core`. Each is published self-contained for win-x64 with `PublishSingleFile=true`, `IncludeNativeLibrariesForSelfExtract=true`, and ReadyToRun, so the release folder contains only `StemForge.exe`, `StemForge.Cli.exe`, and the `tools/` scripts. All managed and native dependencies are embedded in the apphost and self-extracted at runtime. This continues the approach v0.2.0 already used for the GUI and extends it to the CLI; the packaging task zips that folder.

The alternative first recorded here was a self-contained *folder* publish: both executables in one directory sharing a single runtime, which is smaller because the .NET runtime is not duplicated. It was rejected after seeing the artifact. A folder publish scatters ~240 loose runtime DLLs beside the executables, which is a poor release artifact. The hoped-for mitigation, relocating the DLLs into a subfolder, is not cleanly supported for a self-contained app: the apphost requires the native runtime files (`hostfxr`, `hostpolicy`, `coreclr`, `clrjit`, ICU, the Skia/HarfBuzz natives) and the `.deps.json`/`.runtimeconfig.json` to sit next to the executable. Only managed assemblies can be redirected (via `additionalProbingPaths`), which is fragile, hand-edited, and would still leave ~20-30 native/config files at the top level. Single-file is the supported way to get the dependencies out of sight; they are embedded in the exe rather than placed in a subfolder, but the visible result is the clean layout we want.

The cost accepted is size: each single-file executable bundles its own copy of the .NET runtime, so the two duplicate it. Measured on win-x64 with ReadyToRun: the single-file release zips to ~104 MB versus ~60 MB for the shared-runtime folder. The duplicated part is essentially a second .NET runtime, not the full GUI stack, because the CLI carries no Avalonia/Skia. For a desktop application downloaded occasionally, a clean artifact is worth roughly +45 MB. ReadyToRun accounts for much of the bulk (it is the larger reason the CLI single file is ~94 MB); if release size ever needs trimming, dropping ReadyToRun reclaims a substantial amount at the cost of slower cold start, and is the first lever to pull before reconsidering this decision.
