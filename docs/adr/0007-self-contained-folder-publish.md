# Self-contained folder publish instead of single-file

The win-x64 release currently publishes `--self-contained true -p:PublishSingleFile=true`, producing one self-extracting `StemForge.exe` that bundles the entire .NET runtime, Avalonia, and the Skia native libraries. ADR 0006 introduces a second executable (`StemForge.Cli`) alongside the GUI, both sharing `StemForge.Core`. Single-file publishing cannot share a runtime across executables: each single-file exe re-bundles its own copy of the .NET runtime, so shipping two single-file exes would roughly add a second runtime's worth of bytes (on the order of +50% release size) for a CLI that is otherwise mostly the same managed code.

We switch the release to a self-contained *folder* publish. Both executables and one shared set of runtime, Core, and Spectre DLLs live in the same output directory, so the net size growth over today is a couple of MB (essentially Spectre plus a small CLI stub) rather than a duplicated runtime. The release zip already nests everything under a `StemForge/` folder and end users unzip-and-run, so whether that folder holds one `.exe` or two executables beside a set of DLLs makes no difference to their workflow. The packaging task in `.vscode/tasks.json` and any CI packaging step are updated to stage the folder rather than copy a single file.

The cost accepted is that the publish output is no longer a single tidy file — it is an executable plus many loose runtime DLLs, which is the normal shape of a self-contained folder publish but a deliberate reversal of the single-file choice made earlier. This is recorded so the loose-DLL layout is not mistaken for a regression and "tidied" back into single-file, which would silently reintroduce the duplicated-runtime size cost. The single-file route remains available if the size trade-off is ever judged worth it, but it is not the default.
