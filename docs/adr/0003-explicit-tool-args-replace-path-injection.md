# Explicit tool args replace C# PATH injection

StemForge bundles `ffmpeg`, `yt-dlp`, and `deno` into `BundledBinDir` (`%LOCALAPPDATA%\StemForge\bin`). Two of the five tools have indirect runtime dependencies: `yt-dlp` needs a JS runtime (`deno`) to solve YouTube n-challenges, and `audio-separator` needs `ffmpeg` for audio conversion. Neither calls these tools via the StemForge `ProcessRunner`; they look them up via their own PATH search. The original solution was for `ProcessRunner.InjectBundledBinIntoPath` to prepend `BundledBinDir` to every child process's `PATH`, and for `SeparatorDriverService` to duplicate the same injection for the long-lived driver process.

This worked for the common case but broke down once user path overrides were introduced. If a user overrides `ffmpeg` to `C:\MyTools\ffmpeg.exe`, StemForge invokes their binary directly — but `BundledBinDir` is still injected wholesale, so `audio-separator`'s internal PATH search still finds the bundled `ffmpeg`, not the user's. The natural fix (inject the override directory instead) introduces a different hazard: the override directory might contain other executables that shadow bundled tools the user did not intend to override. Alternatives considered — a shim directory of hardlinks, per-tool subdirectories in `BundledBinDir` — were either non-trivial to implement or carried a same-volume constraint on Windows hardlinks.

The actual dependency graph is narrow:

- `yt-dlp → deno`: `yt-dlp` accepts `--js-runtimes deno:/path/to/deno`. Paths with spaces require the value to be wrapped in embedded double quotes (`"deno:path"`) because `yt-dlp`'s own argument parser splits on spaces; the OS-level argument boundary is not sufficient. StemForge emits this arg only when `AppPaths.Deno` resolves to an absolute path (bundled binary present or user override set). When it resolves to a bare name, the arg is omitted and the system PATH is the fallback.
- `audio-separator → ffmpeg`: `audio-separator` exposes no CLI argument for the ffmpeg location. StemForge's `separator_driver.py` accepts `--ffmpeg-path` and prepends its parent directory to `os.environ["PATH"]` before importing `audio-separator`, so the library's internal search finds the right binary. The same absolute-path condition applies: the arg is only passed when the resolved path is absolute.
- `yt-dlp → ffmpeg`: not applicable in StemForge's usage. StemForge uses `yt-dlp` only to resolve a direct media URL, then hands that URL to `ffmpeg` directly. `yt-dlp` never invokes `ffmpeg` in this flow.

With both dependencies covered by explicit args, there is no remaining reason for C# code to inject into `PATH`. `ProcessRunner.InjectBundledBinIntoPath` is removed entirely; the inline injection in `SeparatorDriverService` is removed and replaced by the `--ffmpeg-path` arg at driver spawn time. All `PATH` management is now in Python, scoped to the driver process only.

Known limitation accepted: if a user's override path is absolute but its parent directory happens to contain an executable that conflicts with another bundled tool, the Python driver's `os.environ["PATH"]` prepend for the ffmpeg directory would expose that executable to `audio-separator`. This requires a user to (a) set an ffmpeg override, (b) have a conflicting tool in the same directory, and (c) not want `audio-separator` to use that conflicting tool — a trifecta considered too unlikely to justify a more complex isolation mechanism.
